Problem

ImuApp exposes 20+ fields, mixing connection state, data buffers, and UI flags — interface is nearly as complex as implementation.

Solution

Extract ConnectionManager, DataPipeline, and DeviceController as deep modules with minimal interfaces.

Wins

  • Locality: connection bugs concentrate in one module
  • Leverage: 4 methods replace 20+ exposed fields
  • Testability: mock ConnectionManager without UI
  • Delete 800 lines from main.rs

Unify Serial and BLE Connection Logic

Strong ports & adapters

Files

crates/imu-gui/src/main.rs:270-360 (connect method)

Before: Duplicated connection logic

match connection_type
Serial branch (30 lines):
· create channels
· create SerialTransport
· spawn background task
· store state
BLE branch (35 lines):
· create channels
· get peripheral from manager
· create BleTransport
· connect with peripheral
· spawn background task
· store state
~80% duplicated

After: One adapter per transport

ConnectionManager
connect(device_info):
· match device_info.transport_type
· create appropriate transport adapter
· spawn background task (once)
· store command_tx
Transport adapters: SerialTransport, BleTransport
Both implement Transport trait

Problem

Serial and BLE connection branches duplicate channel creation, task spawning, and state management — adding a third transport type means copying 30 lines.

Solution

ConnectionManager owns the connection logic; transport type is a parameter, not a branch.

Wins

  • Delete 60 lines of duplication
  • Leverage: add transport without touching UI
  • Locality: connection bugs in one place

Worth exploring in-process

Problem

update() is 400 lines, mixing menu rendering with connection logic with plot drawing — hard to find where a UI bug lives.

Solution

Each panel becomes a function in a ui/ module; update() is a thin coordinator.

Wins

  • Locality: UI bugs concentrate in one panel
  • Leverage: reuse panels in other windows
  • update() shrinks from 400 → 20 lines
